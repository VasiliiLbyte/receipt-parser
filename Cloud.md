# Cloud Architecture

## Назначение

Этот документ фиксирует архитектурные принципы проекта receipt-parser.

Он нужен как опорный контекст для любых изменений в коде.
Документ описывает только архитектуру, границы слоёв и обязательные инварианты.

---

## Архитектурная модель

Система рассматривается как единый pipeline обработки чека:

1. приём входного изображения;
2. подготовка изображения;
3. извлечение данных через provider;
4. нормализация результата;
5. валидация и исправление полей;
6. сборка финального структурированного результата;
7. экспорт или сохранение результата.

Главная ценность проекта — не способ запуска, а качество и предсказуемость этого pipeline.

---

## Основные принципы

### Единое ядро обработки

В проекте должно быть одно ядро обработки чека.
Оно должно одинаково работать из CLI, API, worker и тестов.

Ядро не должно зависеть от:
- HTTP;
- очередей;
- конкретного хранилища;
- формата экспорта.

### Модульный монолит

Проект развивается как модульный монолит.
Преждевременное дробление на микросервисы запрещено.

Приоритет:
- простая структура;
- чёткие слои;
- низкая связность;
- минимальная инфраструктурная сложность.

### Изоляция провайдеров

OCR/Vision/LLM provider — это сменяемая внешняя зависимость.
Бизнес-логика не должна зависеть от особенностей конкретного API.

Все провайдеры должны подключаться через единый интерфейс.

### Канонический результат

Внутренний результат обработки — структурированный JSON.
Excel, CSV и другие форматы являются только экспортом.

Это означает:
- pipeline возвращает единый объект;
- валидация работает поверх единой модели;
- экспорт не влияет на доменную логику.

### Обязательная постобработка

Ответ provider не является финальным результатом.
Любые извлечённые данные должны проходить нормализацию и валидацию.

Порядок всегда один:
1. extraction;
2. normalization;
3. validation;
4. result building.

### Прозрачность важнее магии

Логика должна быть явной, тестируемой и воспроизводимой.
Неявные эвристики допустимы только если они локализованы и контролируемы.

## Слои системы

### Input Layer

Отвечает за приём входных данных и приведение их к единому формату.

Не должен:
- выбирать provider;
- валидировать бизнес-поля;
- формировать итоговый результат.

### Processing Core

Центральный orchestration layer проекта.

Отвечает за:
- подготовку изображения;
- вызов provider;
- запуск нормализации;
- запуск валидации;
- сборку результата.

### Provider Layer

Инкапсулирует работу с OpenAI, DeepSeek, локальным OCR и любыми другими движками.

Требования:
- единый интерфейс;
- единый формат ошибок;
- единые таймауты;
- возможность замены provider без изменений в core logic.

Пример контракта:

class ReceiptProvider(Protocol):
    def parse(self, image_bytes: bytes, options: dict | None = None) -> dict:
        ...
Normalization Layer

Приводит извлечённые данные к стабильному внутреннему виду.

Типовые задачи:

очистка строк;

нормализация чисел;

нормализация дат;

унификация имён полей;

устранение OCR-шума.

Validation Layer

Содержит доменные правила проекта.

Типовые задачи:

проверка дат;

проверка сумм;

проверка ИНН;

проверка НДС;

согласование полей;

исправление типовых ошибок распознавания.

Это ключевой слой прикладной ценности проекта.

Result Builder

Собирает финальный результат в единой структуре.
Ни один внешний слой не должен собирать финальный объект вручную из промежуточных данных.

Output Layer

Отвечает только за представление результата.

Примеры:

JSON;

Excel;

CSV;

запись в БД;

передача во внешний сервис.

Output layer не должен содержать OCR, validation или provider logic.

Правила проектирования
Single Responsibility

Каждый модуль выполняет одну роль.
Если модуль делает сразу OCR, validation, export и logging, он должен быть разделён.

Explicit Contracts

Связи между слоями должны быть явными.
Обмен данными должен идти через понятные структуры, а не через неформальные словари.

Provider Isolation

Замена provider не должна ломать validation, result builder или export.

Pure Domain Logic

Доменная логика должна быть максимально чистой:

без сетевых вызовов;

без файловой логики;

без скрытых побочных эффектов;

без зависимости от внешнего runtime.

Testability First

Любой важный слой должен тестироваться изолированно.

Fail Clearly

Ошибки должны быть явными, диагностируемыми и пригодными для логирования.

Формат результата
Финальный результат должен иметь одну и ту же структуру во всех режимах запуска.

Рекомендуемый принцип:

фиксированные верхнеуровневые поля;

отдельный блок метаданных;

отдельный блок предупреждений;

отдельный блок сырого ответа при необходимости.

Пример:

json
{
  "receipt": {},
  "items": [],
  "totals": {},
  "taxes": {},
  "merchant": {},
  "meta": {},
  "warnings": [],
  "raw": {}
}
Структура может уточняться, но канонический формат должен оставаться единым.

Работа с ошибками
Ошибки делятся минимум на четыре класса:

input errors;

provider errors;

validation errors;

system errors.

Требования:

понятная категория;

пригодность для логирования;

пригодность для API-ответа;

сохранение контракта между слоями.

Если данные извлечены частично, система должна по возможности возвращать частичный результат с предупреждениями.

Наблюдаемость
Архитектура должна поддерживать базовую наблюдаемость даже в локальном режиме.

Минимум:

логирование этапов pipeline;

фиксация выбранного provider;

фиксация времени обработки;

фиксация ошибок нормализации и валидации.

Логи должны объяснять, где именно сломался pipeline.

Инварианты
Следующие правила нельзя нарушать без пересмотра этого документа:

Pipeline обработки един для всех способов запуска.

Провайдеры изолированы единым интерфейсом.

Канонический результат существует в одном структурированном формате.

Постобработка и валидация обязательны.

Экспорт отделён от доменной логики.

Бизнес-логика не зависит от конкретного OCR/Vision API.

Инфраструктурная сложность не должна опережать зрелость доменного ядра.
